%%%%%%%%% ABSTRACT
\begin{abstract}
We extend Meta's Animated Drawings system from a clip-based offline animation tool into a webcam- and video-driven motion interface. The original system analyzes a static drawing, builds a 2D rig and textured mesh, and retargets an existing BVH motion clip to deform the drawing. Our contribution adds two motion sources: an offline RGB-video-to-BVH pipeline and a live webcam controller based on MediaPipe pose landmarks. The offline path estimates 13 required body landmarks, repairs missing tracks with non-learned temporal repair and rule-based smoothing, converts normalized landmarks into a MediaPipe-compatible BVH hierarchy, and feeds the result into the existing retargeter. The live path bypasses file generation and directly implements the retargeter interface from a causal MediaPipe stream. We also evaluated a conditional rectified-flow landmark corrector on synthetic BVH-derived corruptions, but leave it disabled in the final system because it does not improve the primary L1 and PCK metrics. The complete prototype supports uploaded drawings, bundled characters, uploaded or recorded motion clips, live camera control, quality reporting, and robust fallback behavior.
\end{abstract}

\section{Introduction}

Traditional character animation separates drawing, rigging, motion capture, retargeting, and rendering into specialist workflows. Meta's \emph{Animated Drawings} project lowers one part of this barrier: a user can upload a humanoid drawing and animate it with a prerecorded BVH motion clip~\cite{Smith2023AnimatedDrawings}. However, the released pipeline is still clip-driven. A child or casual creator cannot simply stand in front of a webcam, wave, and see the drawing wave back.

Our project asks whether a simple RGB camera can become the control signal for this drawing animation pipeline. This is difficult for three reasons. First, monocular human pose estimates are noisy, especially under occlusion, cropping, and fast motion. Second, MediaPipe, BVH files, and the Animated Drawings rig use different coordinate systems, joint names, depth conventions, and skeleton topologies. Third, the live version must be causal: it cannot rely on future frames to clean up the current estimate.

We address these issues by treating motion estimation as an adapter around the existing renderer. The drawing analysis, mesh construction, ARAP deformation, and rendering code remain largely intact. We add a motion layer that maps RGB video or webcam frames into the retargeter contract expected by Animated Drawings. In offline mode, this layer emits standard artifacts: \texttt{pose\_sequence.json}, \texttt{pose\_overlay.mp4}, \texttt{motion.bvh}, and \texttt{motion.yaml}. In live mode, it estimates pose frame by frame and drives the character directly.

\section{Related Work}

\paragraph{Animated drawings.}
The base repository implements Smith et al.'s method for animating children's drawings of human figures~\cite{Smith2023AnimatedDrawings,FacebookAnimatedDrawingsRepo}. Their system detects the drawn figure, segments it, predicts a 2D skeleton, constructs a textured deformable mesh, and retargets human motion to the drawing. We keep this core design and extend the motion input stage.

\paragraph{Human pose estimation.}
We initially considered OpenPose, VideoPose3D, and MocapNET. OpenPose introduced part affinity fields for real-time multi-person 2D pose association~\cite{Cao2017OpenPose}. VideoPose3D shows how temporal convolutions can lift 2D keypoints into temporally consistent 3D pose~\cite{Pavllo2019VideoPose3D}. MocapNET is especially relevant because it predicts BVH-compatible pose from 2D joints~\cite{Qammaz2019MocapNET}. For the implemented prototype, we chose MediaPipe Pose because it is lightweight, easy to run locally, and designed for real-time single-person tracking~\cite{Lugaresi2019MediaPipe,Bazarevsky2020BlazePose,MediaPipePoseDocs}. MediaPipe produces 33 normalized body landmarks; our converter uses the 13 landmarks required by the existing retarget configuration.

\paragraph{Motion representation and deformation.}
BVH is a common skeletal motion format containing a joint hierarchy and per-frame channels~\cite{BlenderBVHDocs,CMUMocap}. Animated Drawings deforms a 2D textured mesh using an as-rigid-as-possible formulation inspired by Igarashi and Igarashi~\cite{Igarashi2009ARAP}. Our work therefore sits at the boundary between camera pose estimation and graphics retargeting.

\section{Method}

\subsection{Original Animated Drawings Pipeline}

The original pipeline has two separable stages. The first stage turns a drawing image into character assets. Given an input image $I$, the code calls TorchServe models for drawn-humanoid detection and drawn pose estimation. It crops the highest-scoring character, builds a foreground mask using adaptive thresholding, flood fill, and contour selection, and writes four main outputs: \texttt{texture.png}, \texttt{mask.png}, \texttt{joint\_overlay.png}, and \texttt{char\_cfg.yaml}. The character configuration stores a 16-joint 2D skeleton with named joints such as \texttt{root}, \texttt{torso}, \texttt{left\_elbow}, and \texttt{right\_foot}.

The second stage retargets a BVH motion clip to this 2D character. Let $X_{t,j}\in\mathbb{R}^3$ be the world-space position of BVH joint $j$ at frame $t$. The original retargeter first normalizes the source skeleton: it rotates the BVH so that the up direction is $+Y$, aligns the estimated forward direction to $+X$, scales the skeleton, and offsets it to a ground plane. For each body-part group $g$, it chooses a projection plane normal $n_g$ either from a fixed frontal/sagittal rule or from PCA over the group's 3D joint trajectories. A character bone orientation is then computed from the projected BVH bone:
\begin{equation}
\alpha_{t,c} =
\angle\left(\Pi_{g}\left(X_{t,j_d} - X_{t,j_p}\right), e_y\right),
\label{eq:retarget-angle}
\end{equation}
where $(j_p,j_d)$ are the proximal and distal BVH joints mapped to character joint $c$, $\Pi_g$ projects onto the selected 2D plane, and $e_y=(0,1)$ is the character's upright reference axis.

The resulting 2D rig drives a textured mesh through ARAP deformation. If $v_i$ and $v_i'$ are original and deformed mesh vertices, $E$ is the edge set, and $h_k$ are handle constraints from the skeleton pins, the deformation can be summarized as
\begin{equation}
\begin{aligned}
E_{\text{arap}}(V',R)=
&\sum_{(i,j)\in E}
\left\|(v_i'-v_j') - R_{ij}(v_i-v_j)\right\|_2^2\\
&+\lambda\sum_k\left\|B_k V' - h_k\right\|_2^2 .
\end{aligned}
\label{eq:arap}
\end{equation}
This objective preserves local edge shape while satisfying the rig handle locations. The renderer then draws the deformed texture with optional depth ordering from the retargeted joints.

\subsection{MediaPipe Video-to-BVH Pipeline}

Our offline extension starts from an RGB video $V=\{I_t\}_{t=1}^{T}$. MediaPipe Pose estimates normalized landmarks
\begin{equation}
P_{t,j}=(x_{t,j},y_{t,j},z_{t,j},s_{t,j}),
\end{equation}
where $s_{t,j}$ is MediaPipe visibility. Although MediaPipe provides 33 landmarks, our converter keeps the 13 required body landmarks: nose, shoulders, elbows, wrists, hips, knees, and ankles.

Low-confidence landmarks are repaired before BVH conversion. For each landmark coordinate, frames with $s_{t,j}<\tau$ are linearly interpolated from visible frames when possible. The sequence is then smoothed by a centered moving average:
\begin{equation}
\bar{p}_{t,j}=\frac{1}{|W_t|}\sum_{u\in W_t}\tilde{p}_{u,j}.
\label{eq:smoothing}
\end{equation}
The postprocessor also checks for likely left-right swaps, sudden hip-center jumps, and foot-contact jitter. If the hip center
\begin{equation}
c_t=\frac{1}{2}\left(\bar{p}_{t,\text{left hip}}+\bar{p}_{t,\text{right hip}}\right)
\end{equation}
moves by more than a threshold, subsequent frames are shifted back toward the previous root trajectory. These operations produce a quality report with detection coverage, repaired landmark ratio, left-right corrections, root-jump corrections, and foot stabilizations.

The BVH converter constructs a compact skeleton with root \texttt{Hip}. For each frame, it defines the image-space hip center $h_t$ and maps landmark $j$ to a BVH-like coordinate
\begin{equation}
Q_{t,j} =
\gamma
\begin{bmatrix}
x_{t,j}-x_{t,h}\\
0\\
y_{t,h}-y_{t,j}
\end{bmatrix},
\label{eq:landmark-to-bvh}
\end{equation}
with scale $\gamma=1000$. This deliberately flattens the body into a frontal plane because the downstream drawing system is primarily 2D. For each controlled BVH bone $b=(j_p,j_d)$ with rest vector $r_b$, the global yaw angle is
\begin{equation}
\theta_{t,b} =
-\operatorname{atan2}
\left(
\det(r_b^{xz}, u_{t,b}^{xz}),
\left\langle r_b^{xz},u_{t,b}^{xz}\right\rangle
\right),
\label{eq:yaw}
\end{equation}
where $u_{t,b}=Q_{t,j_d}-Q_{t,j_p}$. The local rotation written to BVH is the parent-relative difference:
\begin{equation}
\phi_{t,b}=\operatorname{wrap}_{[-180,180]}\left(\theta_{t,b}-\theta_{t,\mathrm{parent}(b)}\right).
\label{eq:local-rotation}
\end{equation}
The converter writes root translation plus \texttt{Zrotation}, \texttt{Xrotation}, and \texttt{Yrotation} channels for the configured joints. The generated \texttt{motion.yaml} points Animated Drawings to this BVH and selects the MediaPipe retarget configuration.

\subsection{Live Webcam Retargeting}

The live webcam mode cannot wait for a full video or write a BVH file each frame. Instead, \texttt{LivePoseRetargeter} implements the subset of the original \texttt{Retargeter} API consumed by \texttt{AnimatedDrawing.update()}: it exposes \texttt{bvh\_joint\_names} and returns orientations, depths, and root position for the current time.

For each webcam frame, \texttt{LiveMediaPipePoseEstimator} produces a \texttt{PoseFrame}. A causal smoother keeps the last good value for invisible landmarks and applies exponential moving average smoothing:
\begin{equation}
\hat{p}_{t,j}=
\begin{cases}
\beta p_{t,j}+(1-\beta)\hat{p}_{t-1,j}, & s_{t,j}\ge \tau,\\
\hat{p}_{t-1,j}, & s_{t,j}<\tau.
\end{cases}
\label{eq:ema}
\end{equation}
The live point conversion uses $(x,1-y,z)$ so that image coordinates align with the character's 2D frame. The live retargeter computes the same character joint orientations as Eq.~\ref{eq:retarget-angle}, with optional root modes: \texttt{locked}, which holds the character at its start location, or \texttt{hip}, which moves the character by the hip-center displacement after a reference frame.

\subsection{Landmark-Flow Ablation}

We also explored, but did not enable by default, a learned temporal postprocessor for unreliable landmarks. The model is a conditional rectified-flow network over 31-frame landmark windows, following the idea of learning a velocity field between paired source and target samples~\cite{Liu2022RectifiedFlow}. Training data is generated from BVH files by forward kinematics, random orthographic projection, and synthetic corruption. Each clean clip is $y\in\mathbb{R}^{T\times J\times2}$ with $T=31$ and $J=13$. The condition is
\begin{equation}
c=\left[x_{\mathrm{corrupt}},\,s,\,m\right],
\end{equation}
where $m_{t,j}=1$ marks corrupted or low-confidence landmarks.

For $t_f\sim U(0,1)$, the rectified-flow state is a straight-line interpolation between corrupt and clean coordinates:
\begin{equation}
y_{t_f}=(1-t_f)x_{\mathrm{corrupt}}+t_f y.
\end{equation}
The target velocity is $u=y-x_{\mathrm{corrupt}}$. A temporal convolutional model $f_\theta(y_{t_f},c,t_f)$ is trained with masked SmoothL1 reconstruction plus a small temporal velocity-smoothness penalty:
\begin{equation}
\begin{aligned}
\mathcal{L}(\theta)=
&\frac{\sum_{t,j}m_{t,j}\rho\left(f_\theta(y_{t_f},c,t_f)_{t,j}-u_{t,j}\right)}
{\sum_{t,j}m_{t,j}+\epsilon}\\
&+\lambda_s\frac{1}{T-1}\sum_{t=1}^{T-1}
\left\|f_{\theta,t+1}-f_{\theta,t}\right\|_1 .
\end{aligned}
\label{eq:flow-loss}
\end{equation}
At inference, Euler integration starts from the corrupt landmarks and applies $K=8$ velocity steps:
\begin{equation}
\hat{y}^{k+1}=\hat{y}^{k}+\frac{1}{K}f_\theta(\hat{y}^{k},c,k/K).
\end{equation}
At runtime, the production pipeline uses non-learned temporal repair and rule-based postprocessing. The flow corrector remains an experimental ablation because its test results did not justify replacing the simpler baseline.

\section{Results}

\subsection{Implemented System}

The repository now contains a complete motion adapter around Animated Drawings. The offline command \texttt{examples/video\_to\_motion.py} converts a short video into pose JSON, overlay video, BVH, and motion config files. The web prototype under \texttt{examples/video\_app} accepts bundled characters, uploaded drawings, uploaded BVH files, and uploaded videos; it scopes jobs by session, reports progress, and returns quality metrics. The live command \texttt{examples/webcam\_to\_animation.py} opens a webcam dashboard, estimates MediaPipe pose frame by frame, smooths the stream causally, and renders the animated drawing next to the camera overlay.

The code submission corresponding to this report is the repository itself. \subsection{Landmark Correction Evaluation}

The learned landmark-flow corrector was trained for 100 epochs with hidden size 128, dropout 0.1, AdamW, 31-frame windows, and 13 landmarks. On the synthetic BVH-derived test split, it did not meet our original acceptance target of improving masked L1 by 10\% over the repair baseline. The final pipeline therefore uses the non-learned repair path. Table~\ref{tab:flow-results} reports the flow model's standalone error profile: its RMSE is reasonable on harder long-span and whole-limb corruptions, but its L1 and PCK are not strong enough to justify enabling it in the deployed system.

\subsection{Technical Discussion}

The main technical lesson is that file compatibility is not the same as motion compatibility. The BVH converter can produce syntactically valid files, but convincing motion depends on coordinate choices, root behavior, joint semantics, and smoothing. A direct 2D-to-BVH conversion is adequate for frontal gestures such as waving, arm raises, and simple stepping, but it cannot recover true out-of-plane rotation from monocular landmarks. MediaPipe's $z$ values are useful as relative cues but are not treated as metric 3D reconstruction in our offline BVH writer.

The live retargeter is often visually more stable than the offline BVH path for immediate interaction because it avoids quantizing motion into a simplified BVH hierarchy. The tradeoff is that live control is less reusable: a saved BVH can be replayed, inspected, and rendered with different characters, while live retargeting is an ephemeral stream.

The learned corrector also clarified the strength of simple baselines. A simple temporal repair baseline is hard to beat when corruptions are short and the underlying motion is smooth, so the final system keeps non-learned repair and rule-based smoothing as the active repair path. The flow model's lower RMSE on long and whole-limb corruptions suggests that learned repair could still help with large failures, but its worse PCK means it often misses the tight 0.02 normalized-coordinate threshold. A practical next version should train a residual model or use learned repair only when confidence diagnostics predict that the baseline will fail.

\subsection{Social Impact}

This project can make animation more accessible to students, hobbyists, and children by turning body movement into an immediate creative input. That democratization is valuable, but it is not impact-free. Easier animation tools may increase the volume of low-cost animated content and could make some pose-based prototyping work less scarce for professional animators. We view the system as a sketching and educational tool rather than a replacement for professional animation judgment, but deployment should avoid implying that automated webcam retargeting substitutes for the artistic labor of rigging, timing, acting, and polish.

Because the interface can use live or recorded video, privacy is the highest-risk issue, especially for younger users. Webcam-driven tools should make local processing clear, avoid unnecessary retention of video, and communicate when camera data leaves the machine. If the system is used by minors, it should require parent or guardian consent, avoid storing identifiable clips by default, and never use a user's video or pose traces for further training without explicit opt-in. Even pose landmarks can be sensitive behavioral data, so exported overlays, pose JSON, and BVH files should be treated as user data rather than harmless metadata.

There is also an accessibility and bias tradeoff. The current interaction assumes a webcam, sufficient physical space, and the ability to make large, camera-visible body movements. People with limited mobility, atypical movement patterns, different body proportions, loose clothing, low lighting, or lower-quality cameras may receive worse pose estimates and less recognizable animations. The system should therefore expose confidence and fallback behavior rather than pretending every body and environment is tracked equally well. A more inclusive version would support prerecorded motion clips, manual pose editing, seated or upper-body modes, and examples that show a wider range of bodies and movement styles.

Finally, although our prototype is not a conversational companion, self-created animated characters can still invite attachment, particularly for children. The tool should be framed as a creative animation workspace, not as a synthetic friend or emotionally responsive partner. If future versions add sharing, character generation, or dialogue, they should include age-appropriate guardrails and avoid design patterns that encourage dependency or parasocial substitution.

\section{Conclusion}

We built a working webcam- and video-driven extension to Animated Drawings. The final system preserves the original drawing analysis and ARAP rendering pipeline while adding MediaPipe motion estimation, temporal repair, BVH conversion, live retargeting, and a web interface. The most reliable contribution is the engineering bridge from camera/video pose to the Animated Drawings retargeter. The landmark-flow model is useful as an experiment and partial hard-case improvement, but our final design keeps it out of the active pipeline until it consistently improves the primary metrics.
